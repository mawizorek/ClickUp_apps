The mechanism

Why one variable can move everything

The magic isn't the slider. It's two decisions working together: semantic tokens, and a perceptual color space.

1. Nothing hardcodes a color

An app never says background: #6b4de6. It says background: var(--accent). The value lives in exactly one place, so there's exactly one place to change it. Ten apps, one dial.

2. Tokens derive, they don't repeat

Every token is computed from the root. --surface-2 is "the background, a bit lighter." --accent-2 is "the accent, rotated 45°." Move the root, the relationships hold.

3. OKLCH keeps it harmonious

OKLCH is oklch(lightness chroma hue), and it's perceptually uniform. Hold lightness + chroma steady and spin only the hue, and every color shifts together without one going muddy or blinding.

tokens.cssthe layer
/* the ONE place a theme is defined */
:root {
  --brand-hue:    285;   /* ← change me */
  --brand-chroma: 0.15;

  /* everything below DERIVES from above */
  --accent:    oklch(74% var(--brand-chroma)
                        var(--brand-hue));
  --accent-2:  oklch(74% var(--brand-chroma)
                        calc(var(--brand-hue) + 45));
  --surface-1: oklch(20% .03 var(--brand-hue));
  --text:      oklch(96% .01 var(--brand-hue));
}

/* every component just points at a token */
.play   { background: var(--accent); }
.art    { background: var(--accent); }
.btn    { background: var(--accent); }
.card   { background: var(--surface-1); }
The relationship

CSS, HTML, and JSON: three jobs, one page

People treat these as "web stuff," but they're three distinct roles. The cleanest apps keep them strictly separated, which is exactly what makes token-swapping possible.

index.htmlstructure

<article class="app">
  <h3 data-bind="title"></h3>
  <p data-bind="metric"></p>
  <button class="btn">
    Upgrade
  </button>

  background: var(--surface-1);
  color: var(--text);
  border-radius: 14px;
}

  background: var(--accent);
  color: var(--on-accent);
}

  "title":  "Studio plan",

How they actually connect

The browser reads HTML and builds a live tree of objects (the DOM). It reads CSS and builds a matching style tree (the CSSOM), then decides how every element looks. JavaScript fetches the JSON and pours the data into the empty HTML slots.

So: HTML is the empty form, JSON is what you write on it, CSS is the ink and paper. Swap the JSON and the content changes. Swap the CSS tokens and the look changes. Neither touches the other.

The one-line takeaway

HTML is the structure. CSS is the style. JSON is the data. Keep them apart, and you can restyle without rewriting, and re-data without redesigning.

The question everyone asks

So what is an .mjs file?

An .mjs file is just JavaScript, but the m stands for "module." It's the modern way to split an app's brain into small, self-contained files that explicitly hand pieces to each other, instead of one giant script where everything is global and order-dependent.

old-school.jsclassic script
// everything is global, load order matters,
// nothing declares what it needs
var theme = "violet";
function applyTheme() { /* ... */ }
// one 4,000-line file. good luck.
theme.mjsmodule
// this file OWNS theming and says so
export function setHue(h) {
  document.documentElement
    .style.setProperty("--brand-hue", h);
}

// another file BORROWS it, explicitly
// app.mjs:
import { setHue } from "./theme.mjs";
setHue(285);

What the module system buys you

import / export. A module publishes only what it wants (export) and pulls in only what it needs (import). No accidental globals, no "which script loaded first" bugs. In the browser you load one with <script type="module" src="app.mjs">.

Why it matters for this repo

This is exactly the direction the app architecture leans: a slim index.html shell that pulls in small, concern-separated modules from source/. MJS is the standard shape of that split, small files each owning one job, wired together by explicit imports. The theme resolver is the poster child: one file exports the resolver, every app imports the same one.

The real world

How an app is built, rendered, and layered

An app is a stack of layers, each one a different concern. Understanding the stack is understanding why token-swapping works: you're editing exactly one layer and leaving the rest untouched.

1
Content HTML
The structure and the words. The DOM tree the browser builds and holds live in memory.
2
Data JSON / API
The living values. Fetched at runtime and poured into the structure. Change it without touching design.
3
Tokens CSS variables
The one dial. Named values (color, spacing, radius) that everything else references.
4
Style CSS
The rules that turn tokens into a look. Selectors, layout, states. Reads tokens, never hardcodes.
5
Behavior JS / MJS
The logic. Handles clicks, fetches data, flips tokens. Split into modules, each owning one job.
6
Render the browser
DOM + CSSOM → render tree → layout → paint → composite. The engine turns all of the above into pixels, and re-paints the moment a token changes.
The through-line: keep the layers separate and each one becomes a dial you can turn on its own. Reskin by editing tokens. Re-content by editing JSON. Re-behave by editing a module. That separation is what lets a whole library of apps share one theme file and reskin, together, from a single variable.
