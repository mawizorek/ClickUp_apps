Where we landed

Question
How do HTML apps connect to a live database?
Answer
A serverless proxy (Cloudflare Worker) in front of the ClickUp API.
Data posture
Read + write · public / non-sensitive, always.
Core rule
Open reads, gated writes (integrity, not secrecy).

Decisions that stick
  • Read and write, public and non-sensitive, always. If sensitive data ever enters, the whole approach is re-scoped, not tweaked.
  • Open reads, gated writes. A public write endpoint needs a server-side integrity gate: shared write-key plus schema validation.
  • Architecture is the ClickUp proxy: a Cloudflare Worker over the ClickUp API. ClickUp stays the single source of truth.
  • One multi-route Worker for the whole Safety Suite, not one per app. Build the plumbing once, ride it for all three forms.
  • Source-of-truth boundary: ClickUp is the public intake front door, FileMaker is the private reporting engine downstream.
  • Sensitive path mapped but not chosen: it flips the front-runner to Supabase with real auth and row-level security.
Where to pick up
  • Pick the pilot: Hazard Assessment for value, or the simplest form for a fastest first end-to-end proof.
  • Define the required-fields contract (schema) for each form before any UI gets drawn.
  • Decide which ClickUp list and fields each form writes into; reuse the existing Risk Assessments and SAFETY Programs structure.
  • Decide where the Worker lives and write the deploy plus secret plus write-key runbook.
  • Design the handoff into the FMP completion-import pipeline.
